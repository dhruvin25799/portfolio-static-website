Git commit -m "and its dilemma"

I'm sure everyone who is on this page has atleast used Git or Github once. While version control is a lifesaver for us clumsy code breaking devlopers. I've often found myself and other developers to be particularly not exactly verbose for their commits, and I get it! You got your code to run after a 1000 edits and now you want me to think of a commit message! But writing small edits isn't going to cut it either. So this blog is for me to come back to and for people who often have one word commit messages.

Why you need good commit messages?

This past one month, I've been working a lot on React and updating my portfolio. Which means a lot of git commits, and also means me breaking my code several times. A good commit message comes handy in here. Let me show you with a bad example. Your code is broken, you need to revert back to a previous state to make sense of it now. That means going through the commit history and reverting back to a point where most of things if not everything was working properly. Now image having to look through a comit history like the image shown below. It's a nightmare right? Checking out and testing every commit to see which one is which. This is the point in time you start wondering if you'd have taken some time to write a meaningful commit message. Well it's never too late to start.

Here are some tips for your future commit messages.

  • Make a multi line commit. A good commit has 3 parts : Subject, body and footer.
  • Limit the subject to 50 characters and be sharp about the changes.
  • The subject is a single line that best sums up the changes made in the commit.
  • Wrap the commit body in 72 characters. This makes developers to look at commit messages on the small terminal.
  • The footer is optional, write future changes to be made, tests to be done or parts to be reviewed here.
